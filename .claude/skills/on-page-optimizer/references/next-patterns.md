# Next.js Page Patterns Reference (ASVAB Hero)

**KEYSTONE doc.** Single source of truth for how on-page SEO edits are made on the ASVAB Hero codebase. Every other skill doc references this one. Read it fully before editing any page.

ASVAB Hero is **Next.js 15 with `output: "export"` (static export)** — there is no SSR, no runtime, no markdown content collection for ranking pages, and no Zod schema. The editable surface for a ranking page is a **TypeScript/JSX `page.tsx` file** under `src/app/`. You edit React, not markdown. Metadata is a typed `export const metadata` object. Body content is hand-authored JSX styled with Tailwind tokens. Structured data is a `<JsonLd>` component. The build emits flat HTML files into `out/`, and the sitemap is generated by a hardcoded script.

This skill targets Next.js only. Ignore everything Astro: no `src/content.config.ts`, no `glob` loaders, no frontmatter, no `.passthrough()`, no `astro-seo`, no trailing-slash config knob.

## 1. URL → file mapping

App Router, file-based. A public URL maps directly to a folder under `src/app/`:

```
src/app/army-asvab-score/page.tsx       ->  /army-asvab-score
src/app/how-to-study-for-the-asvab/page.tsx  ->  /how-to-study-for-the-asvab
src/app/page.tsx                        ->  /            (homepage)
```

The slug **is** the directory name. There is no `slug` frontmatter field, no `getStaticPaths` for these pages, and no filename-vs-frontmatter ambiguity. To find the file for a URL, go to `src/app/{slug}/page.tsx`. To confirm a route exists before linking to it, check that `src/app/{target}/page.tsx` exists.

**Dynamic routes** (`[param]` folders, e.g. `study/[subtest]/[topicSlug]`, `flashcards/[deckSlug]`) are NOT ranking pages and are out of scope — see §11.

## 2. Rendered-title truth (CRITICAL — there is a double-suffix bug)

The root layout `src/app/layout.tsx` defines a title **template**:

```tsx
export const metadata: Metadata = {
  metadataBase: new URL("https://asvabhero.com"),
  title: {
    default: siteTitle,
    template: "%s | ASVAB Hero",   // <-- every child page's title string gets " | ASVAB Hero" appended
  },
  ...
};
```

So a child page that exports `title: "Army ASVAB Scores: ..."` renders in the browser/HTML `<title>` as **`Army ASVAB Scores: ... | ASVAB Hero`** — the source string **+13 characters** (` | ASVAB Hero`).

### ALL title-length / pixel-budget math MUST use the RENDERED string, never the source string.

Source `metadata.title` is `"Army ASVAB Scores: Line Scores, MOS Requirements, GT Guide (2026)"` (64 chars) but Google sees `"... (2026) | ASVAB Hero"` (77 chars). Budget against the rendered value.

### ⚠️ Production double-suffix bug — verify before editing a title

`title: { absolute: "..." }` (the official Next.js way to opt out of the template) is **NOT used on any page in this repo.** Instead, ~31 pages have **manually hardcoded `| ASVAB Hero` into the source title string**, which the template then appends a SECOND time. Confirmed in emitted HTML:

```
src title:  "How to Study for the ASVAB: 8-Step Plan | ASVAB Hero"
out/how-to-study-for-the-asvab.html  ->  <title>How to Study for the ASVAB: 8-Step Plan | ASVAB Hero | ASVAB Hero</title>   ❌ double
out/army-asvab-score.html            ->  <title>Army ASVAB Scores: ... (2026) | ASVAB Hero</title>                          ✅ single
```

**Rule before touching any title:**
1. Read the page's `metadata.title`.
2. If the source string already ends with `| ASVAB Hero` (or any ` | ...` suffix), the rendered title is **double-suffixed** — a real on-page defect. When you optimize that title, **drop the manual suffix** so the template adds it exactly once. Treat this as a fixable SEO issue, not a thing to preserve.
3. If the source string has no suffix, the rendered title = source + ` | ASVAB Hero`. Budget accordingly; do NOT add a manual suffix.
4. There is no `absolute:` override in use. If you ever genuinely need a suffix-free title, use `title: { absolute: "Exact Title With No Brand" }` — but this is rare and should be flagged, not done silently.

> Homepage exception: `src/app/page.tsx` exports a plain-string title and emits with **no** suffix (the root route consumes its own `title.default` rather than the template). Don't generalize from the homepage — child pages always get the template applied.

## 3. Metadata editing

Standard shape on an in-scope page (typed `Metadata`, top of file):

```tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: "Army ASVAB Calculator: Line Scores + Every MOS You Qualify For",
  description:
    "Enter your 9 ASVAB subtest scores. Instantly see your AFQT, all 10 Army line scores ...",
  alternates: {
    canonical: "https://asvabhero.com/army-asvab-calculator",
  },
};
```

- **`title`** — edit per §2 (rendered-string budget; strip duplicate manual suffix).
- **`description`** — plain string. No template applied; what you write is what ships.
- **`alternates.canonical`** — MUST be `https://asvabhero.com/{slug}` with **NO trailing slash**. Never change the canonical to a different slug during an on-page edit; that's a routing change, out of scope.
- Do NOT add OpenGraph/Twitter/robots here unless asked — the root layout already provides site-wide defaults and most pages rely on them.

### `generateMetadata` — the careful path (out of scope by default)

Exactly two pages use `generateMetadata` instead of a static object: `src/app/study/[subtest]/[topicSlug]/page.tsx` and `src/app/flashcards/[deckSlug]/page.tsx`. Both are dynamic, gated, **non-ranking** routes (see §11). The title/description there are computed from data (`frontmatter.title`, `frontmatter.summary`), not hand-written:

```tsx
export async function generateMetadata({ params }: PageProps): Promise<Metadata> {
  const { subtest, topicSlug } = await params;
  const guide = getStudyGuide(subtest, topicSlug);
  if (!guide) return {};
  return {
    title: `${guide.frontmatter.title} — ASVAB ${guide.frontmatter.subtest} Study Guide`,
    description: guide.frontmatter.summary,
    alternates: { canonical: `https://asvabhero.com/study/${subtest}/${topicSlug}` },
  };
}
```

You will not edit these as part of on-page SEO. If a task somehow requires it, treat it as a code change: the title comes from markdown frontmatter in `content/study-guides/`, not from this file.

## 4. H1 + body content (JSX, never markdown)

The H1 lives **in JSX** inside `<article className="prose-asvab">` (~51 article/score-hub pages). Calculator and interactive pages put the H1 in a plain `<div>`/`<header>` instead of `prose-asvab`. There is exactly one H1 per page — do not add a second.

New body sections must be authored as **JSX matching the page's existing patterns and design tokens**. Never insert markdown — there is no markdown renderer for these pages; raw `#`/`##`/`-` would print literally. Copy the surrounding JSX exactly.

### Design tokens used in body JSX (from real pages)

- Text: `text-text-primary`, `text-text-secondary`, `text-text-tertiary`
- Surfaces: `bg-navy`, `bg-navy-light`, `bg-navy-lighter/60`, `bg-accent-dim`
- Borders: `border-navy-border`, `border-navy-border/50`, `border-accent`, `border-accent/30`
- Accent: `text-accent`, `text-accent-hover`, `bg-accent`
- Headings use `font-display`; mono spans use `font-mono`.
- `not-prose` — wrap interactive / card blocks that should escape `prose-asvab` typography (e.g. the related-links `<section>` on the score hub uses `className="my-8 not-prose ..."`).

### Canonical JSX shapes to mimic (verbatim from `army-asvab-score`)

**H1:**
```tsx
<h1 className="font-display text-3xl font-bold text-text-primary sm:text-4xl">

</h1>
```

**H3 subsection:**

**Table (always wrapped in an overflow div):**

```tsx
    <thead>
      <tr className="border-b border-navy-border">
        <th className="pb-2 pr-4 text-left font-semibold text-text-secondary">Category</th>
```

**Aside / callout (Key Point / Tip / Warning) — color signals the kind:**
```tsx

```

**Closing CTA box:**
```tsx
<div className="mt-12 rounded-2xl border border-navy-border bg-navy-light p-6 text-center">
  <h3 className="font-display text-xl font-bold text-text-primary">…</h3>
  <p className="mt-2 text-sm text-text-secondary">…</p>
```

**Escaping:** This is JSX, not markdown. Use HTML entities for typography (`&ndash;`, `&amp;`, `&apos;`, `&ldquo;`/`&rdquo;`) and apostrophes in text (`don&apos;t`), exactly as the existing pages do. A bare `'` inside JSX text is tolerated by the parser but the codebase convention is `&apos;` — match it.

**Calculator/interactive pages** build sections differently: instead of `prose-asvab` they use standalone `<section className="mt-8 rounded-xl border border-navy-border bg-navy-light p-6">` blocks and frequently render rows/FAQs by `.map()`-ing over an inline array (see `army-asvab-calculator` line-score grid and FAQ). When adding content to these pages, mimic the `<section>` card pattern, not `prose-asvab`.

## 5. JSON-LD via `<JsonLd data={...} />`

Component (`src/components/JsonLd.tsx`) is trivial — it stringifies whatever object you pass into a `<script type="application/ld+json">`:

```tsx
export default function JsonLd({ data }: JsonLdProps) {
  return (
    <script type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(data) }} />
  );
}
```

A page can render multiple `<JsonLd>` (e.g. Article + FAQPage). Two authoring variants exist — **preserve whichever the page already uses.**

### Variant (a) — inline object literal (most pages)

`how-to-study-for-the-asvab`, `army-asvab-score`, `army-asvab-calculator`:

```tsx
<JsonLd
  data={{
    "@context": "https://schema.org",
    "@type": "Article",
    headline: "...",
    datePublished: "2026-03-18",
    dateModified: "2026-03-18",
  }}
/>
```

### Variant (b) — extracted const, then referenced (~16 pages)

`navy-asvab-score`, `marines-asvab-score`, `coast-guard-asvab-score`, `bsep`, `afct`, the subtest-tips pages, etc. The object is hoisted to a module-level `const` (named `articleJsonLd`, `faqJsonLd`, …) above the component, then passed by reference:

```tsx
const articleJsonLd = {
  "@context": "https://schema.org",
  "@type": "Article",
  headline: "...",
  datePublished: "2026-03-22",
  dateModified: "2026-05-20",
};
// inside component:
<JsonLd data={articleJsonLd} />
```

When editing freshness or schema on a variant-(b) page, edit the **const**, not a copy inside the JSX.

### Schema type VARIES by page type — never swap it

- Article / score-hub pages → `"@type": "Article"`.
- Calculator/tool pages → `"@type": "WebApplication"` (with `applicationCategory`, `operatingSystem`, `offers`). **Do NOT convert a WebApplication page to Article** or vice versa. `army-asvab-calculator` has no `Article` block by design.
- Interactive practice-test page → `"@type": "Quiz"`.
- FAQ blocks → `"@type": "FAQPage"` (added in addition to the primary type).

Preserve the existing `@type`. Adding new schema types is a content-strategy decision, not a routine on-page edit — flag it.

### Freshness: `dateModified` only

`Article` schema carries both `datePublished` and `dateModified`. When you make a substantive content edit:

- **Update `dateModified`** to today's ISO date (`"YYYY-MM-DD"`).
- **NEVER touch `datePublished`** — it's the original publish date and changing it harms credibility/E-E-A-T.

`WebApplication` and `Quiz` schemas in this repo do not carry date fields — don't add them.

## 6. FAQ sync policy (CRITICAL — duplication is the dominant case)

FAQ content exists in **two places** on most pages: the visible FAQ JSX (what users see) and the `FAQPage` schema's `mainEntity[]` array (what Google reads). They MUST match. There are two patterns — **detect which one the page uses before editing.**

### Pattern A — DUPLICATED (the dominant case, ~61 pages)

The question/answer text is written **twice**: once in `mainEntity[]` and again in the visible JSX. They are NOT linked — editing one does not change the other. Example from `army-asvab-score`:

Schema (top of file):
```tsx
<JsonLd data={{ "@type": "FAQPage", mainEntity: [
  { "@type": "Question",
    name: "What GT score do I need for Army warrant officer?",
    acceptedAnswer: { "@type": "Answer",
      text: "GT 110. No waivers are granted for this threshold. ..." } },
  ...
]}} />
```

Visible JSX (bottom of file):
```tsx
<h3 className="font-display text-base font-bold text-text-primary">
  What GT score do I need for Army warrant officer?
</h3>
<p className="mt-1 text-sm text-text-secondary">
  GT 110. No waivers are granted for this threshold. ...
</p>
```

> Note one allowed difference: the visible JSX may contain `<Link>` markup and `&apos;`/`&ndash;` entities, while the schema `text` is plain prose. The *meaning and wording* must match; the visible version may add inline links and HTML entities.

**RULE for Pattern A:** edit BOTH the visible JSX AND the schema `mainEntity` entry. After editing, re-read both and confirm the question text and answer wording match (modulo links/entities). Adding a new FAQ means adding a `mainEntity[]` object **and** a visible `<h3>`/`<p>` (or `<dt>`/`<dd>`) block. Removing one means removing both.

### Pattern B — SHARED ARRAY (single source of truth, ~1 page)

`free-asvab-practice-test` defines one `faqItems` array and feeds **both** the schema and the visible list from it:

```tsx
const faqItems = [
  { q: "Is this ASVAB practice test really free?", a: "Yes. The 30-question diagnostic ..." },
  ...
];

// schema — generated from the array:
<JsonLd data={{ "@type": "FAQPage",
  mainEntity: faqItems.map((item) => ({
    "@type": "Question", name: item.q,
    acceptedAnswer: { "@type": "Answer", text: item.a },
  })),
}} />

// visible — also rendered from the array:
<dl className="mt-6 space-y-6">
  {faqItems.map((item) => (
    <div key={item.q}>
      <dt className="font-semibold text-text-primary">{item.q}</dt>
      <dd className="mt-2 text-text-secondary">{item.a}</dd>
    </div>
  ))}
</dl>
```

**RULE for Pattern B:** edit the `faqItems` array ONLY. Both surfaces update automatically. Do NOT also hand-edit JSX or schema — you'd create drift.

> Watch out for the calculator look-alike: `army-asvab-calculator` `.map()`s over an **inline** array literal (`[{ q, a }, ...]`) for the *visible* FAQ but its schema (if any) is separate — that is still Pattern A behavior (the array is not shared with a schema generator). The single-source test is: *does the same named array feed both `mainEntity` and the visible list?* Only `free-asvab-practice-test` passes it.

### Detection checklist (run every time before touching FAQs)

1. Find the `FAQPage` block (`grep "FAQPage"`).
2. Is `mainEntity` built from `.map()` over a top-level `const` that is ALSO mapped into the visible list? → **Pattern B**, edit the array only.
3. Otherwise → **Pattern A**, edit both surfaces and verify text parity.

## 8. Internal links

Internal links are `next/link` `<Link>`, used inside prose JSX or component children — never markdown link syntax.

```tsx
import Link from "next/link";
// in body:
<Link href="/calculator" className="text-accent underline hover:text-accent-hover">
  free ASVAB score calculator
</Link>
```

- `href` is the slug with a **leading slash and NO trailing slash**: `/army-asvab-score`, never `/army-asvab-score/`.
- Standard link styling in prose: `className="text-accent underline hover:text-accent-hover"` (some inline links omit `underline`). Match neighbors.
- **Verify the target route exists before adding a link**: `src/app/{target}/page.tsx` must exist (or be a known dynamic route). Do not link to slugs that aren't built — a broken internal link in static export is a 404, not a redirect.
- For navigation-style external anchors the codebase uses plain `<a href ...>` (e.g. fragment links like `href="#test"`, and source links in `VerifiedBlock`). Use `<Link>` for internal route navigation.

To enumerate candidate link targets, list `src/app/*/page.tsx` directories and cross-check against the sitemap whitelist (§10) so you only link to indexable pages.

## 9. Build / dev / verify

Package manager is **npm** (`package-lock.json` present). Scripts (`package.json`):

```json
"dev":   "next dev --turbopack",
"build": "node scripts/generate-sitemap.mjs && next build",
"start": "next start"
```

- **Build:** `npm run build` → regenerates `public/sitemap.xml` THEN runs `next build`, which (because `next.config.ts` sets `output: "export"`) emits a **static export to `out/`**.
- **Dev:** `npm run dev` (Turbopack).

### Verifying a change actually shipped — inspect the emitted HTML, not a dev server

Static export writes **flat files** at the route name (NOT `{slug}/index.html`):

```
/                        ->  out/index.html
/army-asvab-score        ->  out/army-asvab-score.html
/army-asvab-calculator   ->  out/army-asvab-calculator.html
```

(Only the root is `index.html`; every other route is `out/{slug}.html`.) After `npm run build`, confirm your edit landed by grepping the emitted file:

- **Title (catches the §2 double-suffix):** `grep -o "<title>[^<]*</title>" out/{slug}.html`
- **New H2 / FAQ / paragraph text:** `grep "Your new heading text" out/{slug}.html`
- **New internal link:** `grep 'href="/target-slug"' out/{slug}.html`
- **JSON-LD / schema change:** `grep -o 'application/ld+json[^<]*' out/{slug}.html` or grep for the `dateModified` value / a question `name` to confirm it's in the `<script type="application/ld+json">` payload.

If text appears in `page.tsx` but not in `out/{slug}.html`, the build didn't pick it up (didn't rebuild, or the JSX is inside a client component that renders empty in static HTML — check that the content is in a server component, not behind a `"use client"` gate).

## 10. Sitemap / build-artifact policy

`scripts/generate-sitemap.mjs` holds a **hardcoded `pages` array** (~74 entries) and writes `public/sitemap.xml`, stamping **`<lastmod>` with today's date on EVERY build**:

```js
const today = new Date().toISOString().split("T")[0];
const pages = [
  { path: "/", priority: "1.0", changefreq: "weekly" },
  { path: "/army-asvab-score", priority: "0.85", changefreq: "monthly" },
  ...
];
```

Consequences:

- A `public/sitemap.xml` diff that only changes `<lastmod>` dates is **expected build noise**. Do NOT treat it as a dirty/unexpected change, do NOT try to revert it, do NOT flag it.
- **Never hand-edit `public/sitemap.xml`** — it is overwritten on every build. The script is the source of truth.
- Title / description / content / FAQ / link edits do **NOT** touch the sitemap script. Leave it alone.
- The script is edited **only** for slug changes or brand-new pages — and **those are out of scope for this skill** (defer to the `asvab-post-writer` skill, which owns new-page creation + sitemap registration).

The `pages` array doubles as the **indexable-page whitelist** — see §11.

## 11. Page eligibility + what to SKIP

### In scope

Only the indexable ranking pages listed in the `scripts/generate-sitemap.mjs` `pages` array (~74). If a page's slug is not in that array, it is not a ranking target — do not optimize it.

### Out of scope (skip / refuse, even if asked to "optimize" them)

- **Study guides** — `content/study-guides/**` markdown rendered by `src/app/study/[subtest]/[topicSlug]/page.tsx`. These serve the **Pro funnel inside the practice platform, not organic search**: they're a dynamic route, NOT in the sitemap, and use `generateMetadata` driven by markdown frontmatter. The loader (`src/lib/study-guides/loader.ts`) reads `.md` via `gray-matter` + `marked` and renders HTML with `dangerouslySetInnerHTML` inside a `prose prose-invert` (NOT `prose-asvab`) article. **Recognize this model and skip it** — if asked, redirect the user to the `asvab-post-writer` skill, which authors study guides as markdown drops.
